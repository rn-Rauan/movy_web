# Instruções para Lovable — Movy Web

## Stack e Contexto

SaaS de transporte — gerenciamento e reserva de viagens.

**Stack:** React 19 · TanStack Router (file-based) · TanStack React Start · Tailwind CSS · shadcn/ui · Zod · Sonner (toasts)

**API Backend:** variável `VITE_API_URL` — cliente HTTP em `src/lib/api.ts`.

---

## Fluxo Obrigatório Antes de Commit

- Sempre executar `npm run format` antes de preparar commit
- Em seguida, validar `npm run format:check`
- Depois, validar `npm run lint:ci`
- Só commitar quando os 3 comandos passarem sem erro
- Se houver erro de `Delete ␍` (CRLF), normalizar para LF e repetir os comandos

## Rotas (TanStack Router — file-based)

| Arquivo                                    | URL no browser                                                            |
| ------------------------------------------ | ------------------------------------------------------------------------- |
| `index.tsx`                                | `/` — redirect por role                                                   |
| `login.tsx`                                | `/login`                                                                  |
| `signup.tsx`                               | `/signup`                                                                 |
| `public.trip-instances.tsx`                | layout pathless do grupo `/public/trip-instances`                         |
| `public.trip-instances.index.tsx`          | `/public/trip-instances` — marketplace                                    |
| `public.trip-instances.$id.tsx`            | `/public/trip-instances/:id`                                              |
| `public.organizations.$slug.tsx`           | `/public/organizations/:slug`                                             |
| `_protected.tsx`                           | layout pathless — guard central de auth                                   |
| `_protected._admin.tsx`                    | layout pathless — guard de role ADMIN (redireciona não-admins para `/`)   |
| `_protected._admin.drivers.tsx`            | `/drivers` — gestão de motoristas (admin only)                            |
| `_protected._driver.tsx`                   | layout pathless — guard de role DRIVER (redireciona não-drivers para `/`) |
| `_protected._driver.my-trips.tsx`          | `/my-trips` — viagens do motorista (driver only)                          |
| `_protected.my-bookings.tsx`               | `/my-bookings`                                                            |
| `_protected.my-bookings.$bookingId.tsx`    | `/my-bookings/:bookingId`                                                 |
| `_protected.organizations.tsx`             | `/organizations`                                                          |
| `_protected.trips.$orgId.tsx`              | `/trips/:orgId`                                                           |
| `_protected.trips.$orgId.$tripId.tsx`      | `/trips/:orgId/:tripId`                                                   |
| `_protected.trips.$orgId.$tripId.book.tsx` | `/trips/:orgId/:tripId/book`                                              |
| `_protected.setup.tsx`                     | `/setup` — wizard admin 4 passos                                          |

**Convenções críticas:**

- Pathless layouts: prefixo `_` no arquivo (`_protected.tsx`, `_admin.tsx`, `_driver.tsx`)
- `createFileRoute` usa `"/_protected/..."` — URL no browser não tem `_protected`
- Rotas guardadas por role usam flat naming com prefixo pathless: `_protected._admin.drivers.tsx` → URL `/drivers`
- O layout pai (`_admin.tsx` / `_driver.tsx`) precisa ter ao menos uma rota filha — senão TanStack Router trata como rota leaf e gera conflito de URL com `/`
- Após alterar rotas: `routeTree.gen.ts` é regenerado automaticamente em dev
- Guard de auth **apenas** em `_protected.tsx` — não duplicar em rotas filhas

## Autenticação e Roles

**Tokens:** `localStorage` — chaves `tt_access`, `tt_refresh`, `tt_user`

**`useAuth()`** → `{ user, isAuthenticated, loading, login, signup, logout, refreshUser }`

**`useRole()`** → `{ isAdmin, isDriver, adminOrgId, roleLoading, refetchRole }`

- `isAdmin`: user tem role ADMIN na organização
- `isDriver`: user está cadastrado como motorista
- `adminOrgId`: ID da org onde é admin

**`BottomNav`** — tabs por role (precedência: admin > driver > passenger):

- **Passenger:** Explorar · Empresas · Inscrições
- **Driver:** Explorar · Como motorista · Inscrições
- **Admin:** Explorar · Viagens · Configurar

**`index.tsx`** redireciona:

- Não autenticado → `/public/trip-instances`
- Admin → `/_protected/organizations`
- Usuário → `/public/trip-instances`

## Services de API

**NUNCA chamar `api()` direto em rotas ou componentes — sempre usar services.**

```typescript
// trips.service.ts
tripsService.listPublic();
tripsService.listByOrgId(orgId);
tripsService.listBySlug(slug);
tripsService.getPublicById(id);

// bookings.service.ts
bookingsService.listForUser();
bookingsService.getDetails(bookingId);
bookingsService.checkAvailability(tripId);
bookingsService.create({ tripInstanceId, enrollmentType, boardingStop, alightingStop, method });
bookingsService.cancel(bookingId);

// organizations.service.ts
organizationsService.listActive();
organizationsService.listMine();
```

**`ApiError`** — lançado pelo cliente HTTP quando status ≥ 400. Acesse com `err instanceof ApiError`.

## Padrões de Código

### TypeScript

- **Nunca use `any`** — use `unknown` para valores dinâmicos
- Tipos genéricos: `api<T = unknown>(...)` em vez de `T = any`
- Casts apenas quando necessário; nunca `as any` — preferir tipo literal: `v as "ONE_WAY" | "ROUND_TRIP"`
- Novos tipos exportados em `src/lib/types.ts`

### Tratamento de Erros

- `catch (err)` sem tipo
- Acesso: `err instanceof Error ? err.message : "Erro desconhecido"`
- Para erros de API: `err instanceof ApiError`
- Evitar: `catch (err: any)` ou `catch (err as Error)`

### Componentes de Feature

- Recebem dados via **props** — não fazem fetch próprio
- Hooks encapsulam fetch + state + side effects — nunca duplicar lógica entre hooks

### ESLint

- Respeite `eslint.config.js` — não ignore warnings sem justificativa
- `// eslint-disable-next-line` somente com comentário explicativo

---

## O Que NÃO Fazer

- Não chamar `api()` diretamente em rotas ou componentes — usar services
- Não duplicar lógica de fetch — criar/reutilizar hook de feature
- Não modificar `src/components/ui/` — atualizar via CLI do shadcn
- Não duplicar guard de auth — somente em `_protected.tsx`
- Não adicionar React Query — Context API + hooks é suficiente para o MVP
- Não criar OrganizationContext global — `adminOrgId` do `useRole()` é suficiente
- Não alterar arquivos em `/docs/`
